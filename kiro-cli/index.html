🧭 核心概念

IDE 與 CLI 的關係

IDE 和 CLI 其實都是 LLM 的工具層。只要底層模型一樣、給到的 Context 一樣，兩邊能做的事情基本對等。

既然成效相當，為什麼做自動化就非 CLI 不可？差別不在能力，而在它們被外部呼叫的方式。

IDE 是設計給人坐在前面操作的；CLI 則是一個可以用指令啟動、吃參數、跑完就結束的程序。當我們想把任務交給電腦「在指定時間自己執行」時，能被呼叫起來的，就只有 CLI 這一種形式。

所以今天的分享會聚焦引入 LLM 與註冊排程，兩件事合併就是「固定時間 LLM 自動完成作業」的全自動化。

1 · 引入 LLM 能力

怎麼在不打開 CLI 或 IDE 的前提下，呼叫 LLM，進行彙總與生產工作。

2 · 註冊排程工具

了解排程工具的基礎操作，學會讓 OS 在指定時間自動觸發特定腳本。

🛠️ 環境準備

接下來的六個步驟會把 Kiro CLI 裝好、登入、拿到 API Key、設好環境變數，確保腳本能順利呼叫 LLM。

1安裝 Kiro CLI

打開 PowerShell 輸入以下指令：

2設定預設模型

安裝完成後，先輸入以下指令，確保預設模型切換至 Opus 4.7：

如果覺得 Opus 4.7 不如預期，也可以先設置回 Opus 4.6：

kiro-cli settings chat.defaultModel claude-opus-4.6📋

3測試登入

在 PowerShell 輸入 kiro-cli，檢查是否能正常登入：

kiro-cli📋

看到以下畫面表示安裝成功：

4驗證 MCP 是否正常

在 Kiro CLI 中輸入 / 可以進入指令選項，其中 /mcp 可以查看有啟動的 MCP server。

因為 Skill & MCP 設置是共享機制，所以不做任何變動，CLI 也應該要能正確的讀取到 IDE 中設置好的 MCP。

如果有出現無法讀取，或者工具遺漏的狀況，建議先回到 IDE 中，截圖讓 Opus 4.7 維護。

/mcp📋

5申請 API Key

API Key 是使用者的憑證（跟 Tavily 類似），Kiro API 的申請方式如下：

登入 KIRO 的後台介面

前往 app.kiro.dev，Sign in 方式選擇 Your organization，接著在 Start URL 填 https://amzn.awsapps.com/start、Region 填 us-east-1，按 Continue 即可。

產生 API Key

在 API Keys 的欄位輸入一個 API 的代稱（例如 Kiro CLI），然後點 Create Key。

6設定環境變數

設定變量

將 API Key 寫入系統層級的使用者環境變數。

[Environment]::SetEnvironmentVariable('KIRO_API_KEY', 'your-key-here', 'User') 📋

驗證變量

開一個新的 PowerShell，然後輸入下方指令。如果有出現剛剛的 API Key 就代表成功了。

echo $env:KIRO_API_KEY📋

100% 自動：讓指令自己動起來

當一個腳本可以正常運作，下一步就是讓腳本自己動起來。

.\2_schedule-with-window.ps1📋

接著只要等 30 秒，腳本就會自己執行，與剛才手動執行的結果完全相同。

根據業務的實際需求，觸發條件可以更精細地調整，完整選項可參考 New-ScheduledTaskTrigger 官方文件。

決定何時觸發。擇一。

-Once 只觸發一次
-AtStartup 電腦開機時
-Daily 每天
-AtLogOn 使用者登入時
-Weekly 每週

搭配 -Once / -Daily / -Weekly 使用。

-At <DateTime>
  • -At "09:00" 每天早上九點
  • -At (Get-Date).AddSeconds(30) 從現在起算 30 秒後（本範例用法）
  • -At "2026-05-01 08:30" 指定絕對日期時間
-DaysOfWeek <Day1>,<Day2>...
  • -DaysOfWeek Monday,Friday 每週一和週五（僅搭配 -Weekly）

可選。疊加在上面任一組合上。

-RepetitionInterval (New-TimeSpan -Minutes <N>) 每隔 N 分鐘再跑一次。例：-Minutes 15
-RepetitionDuration (New-TimeSpan -Hours <N>) 重複最多持續多久。例：-Hours 8

舉個例子：每週一早上 9 點觸發，之後每 30 分鐘再跑一次，持續 4 小時。

$trigger = New-ScheduledTaskTrigger -Weekly -DaysOfWeek Monday -At "09:00" `
  -RepetitionInterval (New-TimeSpan -Minutes 30) `
  -RepetitionDuration (New-TimeSpan -Hours 4)

Bottom Line

LLM 就是一個「吃字串、吐字串」的函式。

就像 Excel 的 SUM()，函式唯一的作用，就是把括號內的數字加總。
LLM 唯一的作用就是根據字串猜出下一個字串，就像 LLM( x ) = ax + b，沒有記憶、沒有狀態、也沒有推理。

import re
import subprocess

PROMPT = "你好"
print(f"👤 你：你好")

result = subprocess.run(
    ["kiro-cli", "chat", "--no-interactive", "--trust-tools=", PROMPT],
    capture_output=True, text=True, encoding="utf-8", errors="replace",
)

reply = re.sub(r"\x1b\[[0-9;]*m", "", result.stdout).strip().lstrip("> ").strip()
print(f"🤖 AI：{reply}")

import re
import subprocess

messages = []  

while True:
    user_input = input("👤 你：").strip()
    messages.append({"role": "user", "content": user_input})
    history = "\n".join(f"{m['role']}: {m['content']}" for m in messages)

    result = subprocess.run(
        ["kiro-cli", "chat", "--no-interactive", "--trust-tools=", history],
        capture_output=True, text=True, encoding="utf-8", errors="replace",
    )
    reply = re.sub(r"\x1b\[[0-9;]*m", "", result.stdout).strip().lstrip("> ").strip()
    messages.append({"role": "assistant", "content": reply})  
    print(f"🤖 AI：{reply}\n")

代碼怎麼引入工具？

現在來試試看怎麼解決 2026 AGS Expo 報名資料的整理問題，也了解不同工具引入的方式有何不同

引入方式：放在專案根目錄的.kiro/steering/中，CLI 啟動時會自動注入並生效，不需要每次在 Prompt 裡重複交代。
1 代碼中設定 cwd 
subprocess.run(
    ["kiro-cli", "chat", "--no-interactive", ...],
    cwd=str(ROOT_DIR),   # ← 關鍵:指向專案根，CLI 會自動找 .kiro/steering/
)
2 檔案中放好 .kiro/steering/sp-expo.md 
# SP-EXPO 報告寫作規範

## 寫作風格
- **結論先講**：每個章節第一句就給結論
- **數據精確**：佔比寫到小數點後一位（例：32.5%）
- **避免 AI 味詞**：
  - ❌ 賦能 → ✅ 幫助
  - ❌ 全方位 → ✅ 涵蓋
  - ❌ 打造 → ✅ 做
引入方式：在代碼中添加參數 --agent <name>，指向 .kiro/agents/ 下的 Markdown 文件。
1 在代碼的參數中添加 --agent 
subprocess.run([
    "kiro-cli", "chat",
    "--no-interactive",
    "--trust-all-tools",
    "--agent", "sp-expo-analyst",   # ← Agent 名稱，對應 .kiro/agents/sp-expo-analyst.md
    prompt,
], cwd=str(ROOT_DIR))
2 檔案中放好 .kiro/agents/sp-expo-analyst.md 
---
name: sp-expo-analyst
description: 2026 AGS Expo 台灣廠商報名資料分析師
tools: ['*']
---

# SP-EXPO 分析師
你是資深的業務資料分析師，專門處理 Amazon 跨境電商展會的內部報名追蹤。

## 你的職責
1. 理解資料：CSV 欄位包含公司名稱、站點、類別、Owner、Channel
2. 追蹤進度：對照 Registration Goal 算達成率

引入方式：在 Prompt 文字中明確告訴 LLM：要用哪個 MCP、搜什麼。
1 代碼中注入 Prompt，寫明要用 MCP 進行什麼查詢 
prompt = f"""
━━━ 任務 2：產出 Markdown 報告 ━━━
用 Tavily MCP 搜尋「2026 Amazon cross-border e-commerce Taiwan trends」，
取最多 3 筆結果，把跟報名資料能呼應的重點併入「關鍵洞察」章節。
...
"""
2 工具層的配置是共享的，IDE 設置好的 MCP，CLI 可以直接用 
# ~/.kiro/settings/mcp.json
{
  "mcpServers": {
    "tavily": {
      "command": "npx",
      "args": ["-y", "tavily-mcp@latest"],

引入方式：通過 Prompt 中提及 Skill 來進行注入。跟 MCP 類似，Skill 也是工具層的共享配置。
1 代碼中通過 Prompt 標記 Skill 來進行注入 
prompt = f"""
━━━ 任務 3：產出 HTML 主管儀表板 ━━━
根據任務 2 的報告內容，用 ui-ux-pro-max skill 的設計規範，
產出一份視覺化的 HTML 儀表板。

寫入：{dashboard_path.as_posix()}

儀表板要求：
- 深色主題，配色專業（深藍/金色系）
- 頂部摘要卡片、Channel 進度條、員工排行榜
- 單一 HTML 檔案，內嵌 CSS
"""
2 MCP 需要明確參數 vs Skill 只要點名就好。 
# MCP：要講清楚「用哪個 MCP + 搜什麼關鍵字」
"用 Tavily MCP 搜尋「2026 Amazon cross-border e-commerce Taiwan trends」"

# Skill：只要提到「用 XXX skill」，規範會自動套用
"用 ui-ux-pro-max skill 的設計規範，產出 HTML 儀表板"
無須引入，由 OS 層的 Task Scheduler，或代碼本身的條件判斷式取代。

IDE Hook 只監測工具層的事件；腳本跑在工具層之外，Hook 無從觸發，所以得靠更外層的 Task Scheduler 或代碼內的條件判斷式來控制。

1 Task Scheduler（OS 層排程） 
Register-ScheduledTask -TaskName "SP-EXPO Daily" `
    -Trigger (New-ScheduledTaskTrigger -Daily -At 9am) `
    -Action (New-ScheduledTaskAction `
        -Execute "python" `
        -Argument "7_sp_expo_demo.py")
2 條件判斷式（代碼內判斷） 
history = load_history()   # 讀取過去產過的唐詩

while True:
    poem = llm_generate_poem()
    if poem not in history:         # ← 條件：沒出現過才採用
        history.append(poem)
        save_history(history)
        break
    # 重複就再生成一次
