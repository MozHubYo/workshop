從 HTML 到 API
更優雅的數據獲取之道

💡給 non-tech 同事的話

說到「拿一份數據」，大多數人的直覺反應是：

  • 「我去後台點下載按鈕。」
  • 「我請工程師幫我匯出 Excel 傳給我。」
  • 「我一頁一頁複製貼上到試算表。」

這些做法都能工作，但有一個更優雅、也更可持續的方式——直接跟 API 要。

這堂課要讓你看見：你以為要點十下按鈕才能完成的事，用 API 其實一行指令就搞定。而且這個技能比你想像中容易上手。

🧠為何要改變傳統網頁抓取的模式？

HTML 抓取的方法有什麼問題嗎?

我們目前的作法雖然可行，但如同在沙灘上找貝殼，總會撈起一把沙。其存在幾個難以忽視的痛點：

  • 高噪音的信噪比:
    為了獲取個別數據，卻必須載入並處理整個網頁的 HTML，大部分都是我們不需要的「噪音」。
  • 脆弱的維護性:
    腳本與網頁的 UI 結構緊密耦合。任何前端的調整都可能導致腳本失效，造成頻繁的維護成本。
  • 阻塞式的工作流:
    網頁渲染是耗時的。當一個自動化流程需要從多頁面數據時，會造成流程堵塞，無法實現高效的併行處理。

API 數據擷取的優勢

HTML 抓取數據比較像是在河裡淘金，通過獲取來源的所有資料，從中淘取需要的部分。
相較之下，API 更像是直接從金庫取金條，拿出來的數據就是純金，幾乎沒有雜質。

API 的三大優勢
  • 精準獲取: API 允許用戶只請求所需的數據，將「噪音」降至趨近於零。
  • 高度穩定: API 和數據結構極少變動。一次開發便可長期穩定運行，維護成本極低。
  • 可擴展的基建: API 請求是輕量級操作，可以輕易地併行處理，即使工作流持續擴張，仍可保持高效率。

🚀 API 的初體驗

爬取頁面數據 v.s. 訪問 API 的工作流對比

假設業務目標是：抓取 直播 與 線下課程 的最新課程列表。

傳統頁面抓取方法

  1. 進入目標網頁。
  2. 等待網頁渲染。
  3. 解析頁面上的 HTML 元素來提取課程標題

API 導向方法

  1. 監控 "Network" 查找 API 資料。
  2. 使用 fetch()發出模擬訪問 API 的請求。
  3. 獲取數據。

實作體驗：第一次 API 呼叫

我們用一個公開的免費 API 來示範——JSONPlaceholder（專門給開發者練習的假資料 API，不用註冊、不用 Key）。

打開任何一個網頁，按 F12 開啟主控台（Console），貼上以下代碼：

// 目標 API：拿一筆貼文的資料
fetch('https://jsonplaceholder.typicode.com/posts/1')
  .then(response => response.json())
  .then(data => {
    console.log(data);
  });

你會看到主控台出現一個結構清晰的物件，包含 userId、id、title、body 四個欄位。

接著把網址換成 /posts（不加 id），會回傳 100 筆貼文的陣列。這整個過程你沒有載入任何頁面，沒有解析 HTML，但你拿到的資料比手動爬頁面還乾淨。

💡通過 API 抓取數據，就不需要等待網頁渲染，也不需要管 HTML 結構 我們並未實際「載入」任何頁面，卻可以直接拿到結構化的資料。這就是 API 與爬取 HTML 兩種作法上的本質差異。

探索邊界：API 的保護機制

剛才的 JSONPlaceholder 是完全開放的，任何人都能呼叫。但真實世界的 API 通常有保護機制。當你嘗試從別的網域呼叫受保護的 API 時，會在主控台看到類似這樣的紅色錯誤訊息：

Access to fetch at 'https://some-site.com/api' from origin 'https://another-site.com' has been blocked by CORS policy...

這代表 API 的使用並非無條件，尤其越敏感的資料，通常也會有越麻煩的保護機制，常見的保護機制包括：

  • 同源策略 (CORS) : 資料的請求必須來自於同一個網域，限制/禁止跨網域的請求。
  • Cookie / Session : 驗證用戶是否為合法的使用者，通常會搭配登入機制。
  • Token / API Key : 類似權限證明，確保用戶有權限進行操作，或者與付費機制綁定。
通過 API 獲取數據，並非永遠都是最佳解 在 API 端口所要求的參數與環境過於複雜時，深度地去破解這些參數本身可能更花時間。
這種情況下，可能使用傳統的 HTML 數據抓取方式，會比使用 API 簡單很多。

核心觀念總結

學習的目標不是 拋棄HTML資料爬取 的做法，而是 增加工具箱內的選項 。
根據不同的業務場景，選擇更加合適與高效的解決方案。
了解用戶不僅可以「在渲染好的頁面上尋找數據」，也可以「向特定的 API 端點發出數據請求」。

🛠️上手 fetch() 與 API 偵測

fetch() 語法解析

fetch() 的結構相當直觀，其運作方式類似一條流水線：

fetch("https://example.com/api/data.json") // 1. 發出請求
  .then(response => response.json())       // 2. 將回傳的數據流轉換為 JSON 物件
  .then(data => {                          // 3. 使用處理好的數據
    console.log(data);
  });

若想深入了解，可參考 MDN Fetch API 的官方文件。



整個API取數的過程，最關鍵的步驟就2點： 1) 找出正確的 API 端口； 2) 找出端口要求的參數：

如何在 Network 面板中找到 API 相關資料？

  1. 打開開發者工具 (F12)，切換到 Network 分頁。
  2. 在頁面上觸發目標操作 (例如：點擊搜尋、切換分頁)。
  3. 篩選網路請求: 通常 API 請求的類型是 fetch 或 XHR，點擊它可以篩選出可能的 API 請求。
  4. 分析目標請求: 點擊列表中的請求，右側會出現詳細資訊。
    • Headers: 查看請求的 URL、方法 (GET/POST)、以及是否有特殊的 Authorization 或 Token 標頭。
    • Payload / Body: 如果是 POST 請求，這裡會顯示發送給伺服器的數據結構。
    • Preview / Response: 檢視伺服器回傳的數據格式，確認其為您所需的數據。

上手練習：在你平常用的網站上找 API

打開任何一個你常用的網站（例如購物、新聞、查股票），F12 切到 Network 分頁，觀察你操作時實際發出了哪些 API 請求。

// 步驟 1：使用 Network 觀察窗，找到正確的 API 端口，通常會像一個網址
"https://example.com/api/data"

// 步驟 2：確定 Headers / Token / Payload / Body 後，用 fetch() 發出請求
fetch("https://example.com/api/data")
  .then(response => response.json());

// 步驟 3：拿到資料後，作必要的清洗與加工
    console.log(data);

🏆課堂挑戰：三道漸進式實戰題

前面都是概念，現在真的動手。我們會用 public-apis（GitHub 上收錄了上千個公開 API 的清單）裡挑出來的三個 API，通通不用註冊、不用 Key、不用 Token，貼上就能跑。

操作方式：打開瀏覽器主控台（任何網頁 F12 都行），把代碼貼進去按 Enter。

🟢 第一題｜最基本的 GET 請求

目標：取得一張隨機的狗狗圖片。

API：Dog CEO API（dog.ceo）

fetch('https://dog.ceo/api/breeds/image/random')
  .then(response => response.json())
  .then(data => {
    console.log('狗狗圖片網址：', data.message);
  });
💡 思考一下 如果沒有這個 API，你要怎麼「隨機」拿到一張狗狗圖？去 Google 圖片搜尋、手動點一張、右鍵複製網址？用 API 一行就搞定，而且每次執行結果都不同。

🟡 第二題｜帶參數的查詢

目標：查詢台北當下的氣溫。

API：Open-Meteo（open-meteo.com，公開免費的天氣 API）

// latitude=緯度, longitude=經度（台北的座標）
const url = 'https://api.open-meteo.com/v1/forecast?latitude=25.03&longitude=121.56¤t=temperature_2m';

fetch(url)
  .then(response => response.json())
  .then(data => {
    const temp = data.current.temperature_2m;
    console.log(`台北現在 ${temp}°C`);
  });
💡 思考一下 原本查天氣你要做什麼？打開中央氣象局網站 → 找到台北 → 看溫度。現在你把這段代碼塞進一個排程腳本，每天早上自動回報溫度給你。這才是「自動化」真正的樣子。

🔴 第三題｜多個 API 組合（這題才展現 API 的威力）

目標：給一個國家名稱，自動回傳它的首都、人口，還有首都現在的天氣。

用到的 API：

  • REST Countries（restcountries.com）— 國家基本資料
  • Open-Meteo — 天氣資料
async function getCountryInfo(countryName) {
  // Step 1：拿國家資料
  const countryRes = await fetch(`https://restcountries.com/v3.1/name/${countryName}`);
  const countries = await countryRes.json();
  const country = countries[0];

  const capital = country.capital[0];
  const population = country.population.toLocaleString();
  const [lat, lon] = country.capitalInfo.latlng;

  // Step 2：用首都座標查天氣
  const weatherRes = await fetch(
    `https://api.open-meteo.com/v1/forecast?latitude=${lat}&longitude=${lon}¤t=temperature_2m`
  );
  const weather = await weatherRes.json();
  const temp = weather.current.temperature_2m;

  console.log(`📍 ${country.name.common}`);
  console.log(`   首都：${capital}`);
  console.log(`   人口：${population}`);
  console.log(`   首都氣溫：${temp}°C`);
}

getCountryInfo('Japan');
🎯 核心重點 這一題示範了 API 最強大的能力——組合。第一個 API 的輸出（首都座標）變成第二個 API 的輸入（查天氣）。
如果用傳統方式，你要查日本首都、再查東京天氣，至少要切換 3 個分頁、點 10 次滑鼠。而這段代碼把整個流程壓縮成一個函式，還能換成任何國家重複執行。
